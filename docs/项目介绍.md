# 项目介绍

## 1. 项目名称

宠物急救与寻宠协同平台

## 2. 项目定位

本项目是一个面向养宠用户的多端产品，重点解决宠物突发情况、宠物走失、宠物档案管理和本地宠物服务信息分散的问题。

第一阶段不做综合宠物电商，也不做复杂医疗诊断平台，而是聚焦以下场景：

- 宠物出现异常情况时，用户能快速查看急救知识；
- 用户能快速找到附近宠物医院并拨打电话；
- 宠物走失时，用户能快速发布寻宠信息；
- 用户能建立宠物档案，便于日常管理和紧急联系；
- 平台运营方能在后台审核和维护核心内容。

## 3. 登录产品策略

用户端采用 **游客模式 + 渐进式登录**。

用户首次进入应用时，不强制登录。公开内容允许游客直接浏览。

游客可访问：

- 首页；
- 急救知识；
- 宠物医院；
- 寻宠列表与详情；
- 领养列表与详情；
- 社区帖子列表与详情；
- H5 分享页；
- 用户协议与隐私政策。

需要登录的行为：

- 创建宠物档案；
- 编辑宠物档案；
- 发布寻宠；
- 发布领养；
- 发布帖子；
- 评论；
- 点赞；
- 收藏；
- 上传图片；
- 查看我的宠物；
- 查看我的发布；
- 查看我的收藏；
- 修改个人资料。

管理后台必须登录，且需要管理员权限。

## 4. 目标用户

| 用户类型 | 说明 |
|---|---|
| 普通养宠用户 | 浏览急救知识、医院信息、寻宠和社区内容 |
| 注册用户 | 管理宠物档案、发布内容、收藏、评论 |
| 丢宠用户 | 发布寻宠信息，生成传播内容 |
| 宠物医院 / 宠物店 | 后期可被收录为本地服务资源 |
| 平台管理员 | 审核内容、维护医院数据、管理用户和基础信息 |

## 5. 核心问题

当前养宠用户在突发场景中常见问题：

1. 宠物突发异常时，不知道先做什么；
2. 附近宠物医院信息分散；
3. 宠物走失后，寻宠内容制作和传播效率低；
4. 宠物档案分散，紧急情况下难以快速提供关键信息；
5. 社区内容需要审核和治理。

## 6. 产品目标

### 6.1 第一阶段目标

- 完成用户端基础功能；
- 完成后端 API；
- 完成管理后台；
- 完成数据库、缓存和文件存储基础能力；
- 支持公开浏览和渐进式登录；
- 支持内容审核和基础运营管理。

### 6.2 中期目标

- 扩展本地宠物医院和宠物店数据；
- 优化寻宠传播链路；
- 增加宠物健康提醒；
- 提升后台运营能力；
- 建立真实用户反馈闭环。

### 6.3 后期目标

- 接入服务预约；
- 支持商家入驻；
- 增加宠物服务推荐；
- 建立城市级本地宠物服务网络。

## 7. MVP 功能范围

| 模块 | 功能 |
|---|---|
| 首页 | 急救入口、医院入口、寻宠入口、宠物档案入口 |
| 用户模块 | 登录、用户资料、Token 管理 |
| 宠物档案 | 新增宠物、编辑宠物、查看宠物详情 |
| 急救知识 | 急救分类、急救文章、收藏 |
| 宠物医院 | 医院列表、医院详情、电话、地址、地图位置 |
| 寻宠模块 | 发布寻宠、查看寻宠、上传图片 |
| 领养模块 | 发布领养、查看领养 |
| 社区模块 | 发帖、评论、点赞、收藏 |
| 文件上传 | 图片上传、文件 URL 返回 |
| 管理后台 | 内容审核、医院管理、用户管理 |

## 8. 产品边界

本项目可以提供急救知识、风险提示、医院信息和用户记录能力，但不应宣称替代兽医诊断。涉及严重症状时，应引导用户尽快联系线下宠物医院。

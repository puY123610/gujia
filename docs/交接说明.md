# 交接说明

## 1. 当前交接状态

当前项目已经收口到“可交接开发骨架”阶段。

已完成：

- 根目录 pnpm workspace；
- 用户端、后端、管理端三端基础目录；
- OpenAPI 首版权限分层契约；
- Postman 集合基线；
- Docker Compose 服务编排骨架；
- 环境检查、初始化、类型生成、启动、停止、部署、备份脚本；
- Public / UserAuth / AdminAuth 权限边界文档；
- UserAuth / AdminAuth 后端守卫默认不放行。

未完成：

- 真实用户登录和 JWT 签发；
- Prisma 业务模型和迁移；
- 各业务模块 Controller / Service / DTO / VO；
- 前端页面业务交互；
- 管理后台路由、登录和审核页面；
- 文件上传真实存储逻辑；
- 自动化测试和 CI。

## 2. 新开发者进入项目

从项目根目录执行：

```bash
pnpm check:env
pnpm install
pnpm services:start
pnpm generate:types
```

启动开发服务：

```bash
pnpm dev:backend
pnpm dev:frontend:h5
pnpm dev:admin
```

只检查骨架、不安装依赖：

```bash
SKIP_INSTALL=1 bash scripts/init-project.sh
```

## 3. 分支协作方式

本项目当前使用以下分支：

| 分支 | 用途 |
|---|---|
| `main` | 稳定分支，只放测试通过的代码 |
| `dev` | 联调测试分支，所有功能先合到这里 |
| `feat/frontend-pet-uniapp` | 前端开发分支 |
| `feat/backend-pet-nestjs` | 后端开发分支 |

所有人开发前先确认自己所在分支：

```bash
git status --short --branch
```

Owner 如果写前端：


## 4. 开发顺序建议

1. 后端先实现 Auth 模块，完成 JWT 签发、解析和用户身份注入。
2. Prisma 先落用户、管理员、宠物档案、医院、急救文章、寻宠、文件资源核心模型。
3. 先实现 Public 接口，确保游客模式可用。
4. 再实现 UserAuth 接口，所有用户数据从 Token 解析 userId。
5. 最后实现 AdminAuth 审核和管理接口。
6. 前端和管理端严格按 `api-contract/openapi.yaml` 对齐字段。

## 5. 权限边界

Public：

- 不需要 Token；
- 不得因为未登录阻断首页、急救知识、医院列表、寻宠详情等公开内容。

UserAuth：

- 必须校验普通用户 Token；
- 后端不得相信前端传来的 userId；
- userId 必须从 Token 解析。

AdminAuth：

- 必须校验管理员 Token；
- 必须校验管理员权限；
- 不允许普通用户访问管理端接口。

## 6. 变更同步要求

新增或修改接口：

- 更新 `docs/接口文档.md`；
- 更新 `api-contract/openapi.yaml`；
- 重新生成 `api-contract/generated-types/api-types.ts`。

修改数据库：

- 更新 `docs/数据库设计.md`；
- 增加 Prisma migration；
- 标明是否需要数据迁移或回滚。

修改登录策略：

- 更新 `README.md`；
- 更新 `docs/项目介绍.md`；
- 更新 `docs/技术架构说明.md`；
- 更新 `AGENTE.md`。

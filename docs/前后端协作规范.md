# 前后端协作规范

## 1. 协作原则

1. 接口先设计，再开发。
2. 字段先约定，再联调。
3. 每个接口必须标记权限等级。
4. 后端不能随意改返回结构。
5. 前端不能私自猜测字段。
6. 接口变更必须同步文档。
7. 数据库变更必须通知前端影响范围。
8. 联调问题必须记录，不靠口头记忆。

## 2. 权限协作规则

统一使用三类权限：

| 权限等级 | 说明 | 前端处理 |
|---|---|---|
| Public | 游客可访问 | 直接访问 |
| UserAuth | 用户登录后访问 | 未登录时弹出登录引导 |
| AdminAuth | 管理员访问 | 未登录或无权限时跳转管理端登录页 |

前端不能把 Public 页面做成启动即登录。后端不能把 Public 接口错误地加上强制登录守卫。

## 3. 登录交互规则

用户端登录逻辑：

```txt
用户浏览公开内容
↓
点击需要登录的功能
↓
弹出登录引导
↓
用户登录
↓
回到原页面或继续原操作
```

管理端登录逻辑：

```txt
访问管理后台
↓
检查管理员 Token
↓
无权限则跳转登录页
↓
登录成功后进入后台
```

## 4. 角色职责

| 角色 | 职责 |
|---|---|
| 产品 / 项目推进 | 确认需求、功能边界、优先级 |
| 前端 | 页面实现、接口调用、类型定义、平台适配 |
| 后端 | API、业务逻辑、数据存储、权限、安全 |
| 管理端 | 审核、运营、基础数据管理 |
| 测试 / 自测 | 验证功能、记录问题、回归测试 |

## 5. 接口设计流程

```txt
确认需求
↓
确认权限等级
↓
设计接口字段
↓
写入接口文档
↓
后端实现接口
↓
前端根据文档调用
↓
联调
↓
修复问题
↓
确认完成
```

## 6. 字段命名规范

接口字段统一使用小驼峰：

```json
{
  "userId": "1",
  "petName": "豆豆",
  "avatarUrl": "https://example.com/pet.jpg"
}
```

数据库字段可以根据 ORM 规则映射，但接口层必须保持统一。

## 7. 统一返回格式

所有接口返回：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

前端只根据统一结构处理响应，不为每个接口单独写不同解析逻辑。

## 8. 错误处理规范

| code | 前端处理 |
|---|---|
| 400 | 提示参数错误 |
| 401 | 清除登录状态，需要时引导登录 |
| 403 | 提示无权限 |
| 404 | 提示数据不存在 |
| 500 | 提示服务异常 |

Public 页面遇到 401 时，需要检查是否错误调用了需要登录的接口，或后端是否错误配置权限。

## 9. 文件上传协作

文件上传统一走上传接口。

流程：

```txt
前端选择文件
↓
若未登录，引导登录
↓
调用上传接口
↓
后端校验文件
↓
后端上传到文件存储
↓
返回 fileUrl
↓
前端提交业务表单
↓
业务表保存 fileUrl
```

前端不要直接把文件上传到对象存储，除非后期明确设计了直传方案。

## 10. 分页规范

列表接口建议统一分页参数：

| 参数 | 类型 | 说明 |
|---|---|---|
| page | number | 页码，从 1 开始 |
| pageSize | number | 每页数量 |

分页返回建议：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "list": [],
    "page": 1,
    "pageSize": 10,
    "total": 100
  }
}
```

## 11. 审核状态规范

涉及用户发布内容的模块，应考虑审核状态：

| 状态 | 说明 |
|---|---|
| pending | 待审核 |
| approved | 已通过 |
| rejected | 已拒绝 |

适用模块：

- 寻宠；
- 领养；
- 社区帖子；
- 评论，视情况；
- 医院信息，视情况。

## 12. 联调要求

联调前：

1. 后端确认接口已完成；
2. 后端用接口测试工具验证；
3. 前端确认字段和类型；
4. 双方确认接口权限等级；
5. 双方确认环境地址；
6. 明确测试账号和测试数据。

联调中：

1. 问题必须记录；
2. 明确是前端问题、后端问题还是需求问题；
3. 字段变更必须更新文档；
4. 权限等级变更必须同步前后端；
5. 修复后必须回归测试。

## 13. 变更规则

### 13.1 新增接口

必须更新：

- `docs/接口文档.md`
- `api-contract/openapi.yaml`
- 前端 `api/`
- 前端 `types/`

### 13.2 修改接口字段

必须说明：

- 修改了哪个接口；
- 修改了哪个字段；
- 是否兼容旧字段；
- 前端需要修改哪些页面。

### 13.3 修改权限等级

必须说明：

- 接口从什么权限变成什么权限；
- 是否影响游客访问；
- 是否影响登录跳转；
- 是否影响管理端权限。

### 13.4 修改数据库

必须说明：

- 修改了哪些表；
- 是否需要迁移；
- 是否影响旧数据；
- 是否影响接口返回。

## 14. 验收标准

一个功能完成，至少满足：

1. 页面可正常打开；
2. Public 页面未登录可访问；
3. UserAuth 操作未登录时能正确引导登录；
4. 登录成功后能返回原页面或继续原操作；
5. 表单校验正常；
6. 接口请求正常；
7. 成功和失败状态都有提示；
8. 数据能正确保存；
9. 刷新或重新进入后数据仍存在；
10. 权限错误能正确处理；
11. 文档已同步更新。

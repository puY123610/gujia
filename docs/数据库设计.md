# 数据库设计

## 1. 数据库选型

本项目使用 PostgreSQL 作为主数据库。涉及地理位置能力时使用 PostGIS 扩展。

数据库职责：

- 保存用户、宠物、医院、寻宠、领养、社区等长期业务数据；
- 保存文件资源的元信息和访问地址；
- 支持附近医院、寻宠位置等空间查询；
- 不保存图片、视频、附件本体。

## 2. 登录策略对数据库的影响

用户端支持游客模式，因此公开内容不依赖用户登录。

数据库设计需要区分：

| 数据类型 | 是否必须关联用户 |
|---|---|
| 急救知识 | 否 |
| 医院信息 | 否 |
| 寻宠公开列表 | 发布时需要用户，浏览时不需要 |
| 领养公开列表 | 发布时需要用户，浏览时不需要 |
| 社区帖子 | 发布时需要用户，浏览时不需要 |
| 宠物档案 | 必须关联用户 |
| 收藏、评论、点赞 | 必须关联用户 |
| 管理后台数据 | 必须关联管理员或操作记录 |

## 3. 通用字段

核心业务表建议包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | UUID / BigInt | 主键 |
| createdAt | timestamp | 创建时间 |
| updatedAt | timestamp | 更新时间 |
| deletedAt | timestamp | 软删除时间，可选 |
| status | string / enum | 状态字段，可选 |

## 6. 文件存储原则

图片、视频、附件本体不存入 PostgreSQL。

正确方式：

```txt
文件本体 → MinIO / OSS / COS
文件 URL → PostgreSQL
```

## 7. 变更规则

1. 数据库结构变更必须使用迁移文件。
2. 修改表字段必须同步更新接口文档。
3. 修改字段类型必须同步更新前端类型。
4. 生产环境禁止直接手工改表。

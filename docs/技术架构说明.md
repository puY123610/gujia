# 技术架构说明

## 1. 总体架构

本项目采用前后端分离架构，分为用户端、管理端、后端服务、数据库、缓存、文件存储和部署网关。

```txt
微信小程序 / Android App / H5
        ↓
uni-app 用户端
        ↓
NestJS API 服务
        ↓
PostgreSQL / Redis或Valkey / 文件存储
        ↓
Nginx + Docker Compose 运行环境
```

## 2. 登录与权限架构

用户端采用游客模式和渐进式登录。

权限分为三层：

| 权限等级 | 含义 | 示例 |
|---|---|---|
| Public | 游客可访问 | 首页、急救知识、医院列表、寻宠详情 |
| UserAuth | 普通用户登录后访问 | 发布、收藏、评论、宠物档案 |
| AdminAuth | 管理员登录后访问 | 内容审核、用户管理、医院管理 |

### 2.1 用户端访问逻辑

```txt
用户打开应用
↓
直接进入首页
↓
浏览公开内容
↓
点击需要身份的功能
↓
弹出登录引导
↓
登录成功
↓
继续原本操作
```

### 2.2 管理端访问逻辑

```txt
访问管理后台
↓
检查管理员 Token
↓
无 Token 或权限不足
↓
跳转管理员登录页
↓
登录成功后进入后台
```

## 3. 技术选型

| 层级 | 技术 | 作用 |
|---|---|---|
| 用户端 | uni-app + Vue3 + TypeScript | 多端页面、用户交互、接口调用 |
| 管理端 | Vue3 + TypeScript | 内容审核、数据管理、运营管理 |
| 后端 | NestJS + TypeScript | API、业务逻辑、权限校验 |
| 数据库 | PostgreSQL | 保存长期结构化业务数据 |
| 地理位置 | PostGIS | 支持附近医院、寻宠位置等空间查询 |
| 缓存 | Redis / Valkey | 缓存、验证码、限流、临时锁 |
| 文件存储 | MinIO / OSS / COS | 保存图片、视频、附件 |
| 部署 | Docker Compose + Nginx | 服务编排、反向代理、HTTPS |

## 6. 数据库架构

数据库使用 PostgreSQL。空间位置相关数据使用 PostGIS 扩展。

主要保存：

- 用户信息；
- 宠物档案；
- 急救知识；
- 医院信息；
- 寻宠信息；
- 领养信息；
- 社区帖子；
- 评论收藏；
- 文件资源；
- 管理员数据。

数据库只保存文件地址和元信息，不保存图片、视频本体。

## 7. 缓存架构

Redis / Valkey 用于缓存和临时数据。

典型使用：

- 验证码；
- Token 黑名单；
- 热门急救文章；
- 热门寻宠列表；
- 附近医院查询缓存；
- 接口限流；
- 临时锁。

缓存不是主数据库，缓存失效后必须能从 PostgreSQL 恢复数据。

## 8. 文件存储架构

开发环境建议使用 MinIO。生产环境可根据成本和团队资源选择阿里云 OSS、腾讯云 COS 或自建 MinIO。

上传流程：

```txt
用户选择图片
↓
前端调用上传接口
↓
后端校验文件
↓
上传到文件存储
↓
返回文件 URL
↓
业务表保存文件 URL
```

## 9. 部署架构

第一阶段使用 Docker Compose 部署：

- PostgreSQL；
- Redis / Valkey；
- MinIO；
- NestJS 后端；
- Nginx；
- 管理端静态资源。

Nginx 负责：

- HTTPS；
- API 反向代理；
- H5 静态资源；
- 管理端静态资源；
- 基础访问控制。

## 10. 架构原则

1. 用户端、管理端、后端服务分离。
2. 用户端先展示价值，再引导登录。
3. 管理端必须登录。
4. 接口按 Public、UserAuth、AdminAuth 分层。
5. 数据库结构通过迁移管理。
6. 文件存储和数据库职责分离。
7. 缓存可失效，核心数据不能丢。
